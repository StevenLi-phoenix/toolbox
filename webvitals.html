Official thresholds (75th percentile of page loads)

A metric is Good only when the 75th-percentile value across mobile and desktop hits the Good threshold. Above the Poor cutoff it's Poor; in between it's Needs Improvement. These thresholds come from web.dev/vitals (March 2024 INP migration).

Notes & FAQs

  • Why the 75th percentile? A site passes a Core Web Vital only when 75% of real-user visits hit the Good threshold. A median ("typical") visit doing fine isn't enough — the long tail counts.
  • Lab vs field. Lighthouse (lab) gives you a single number; CrUX (field) gives you the percentile of your real users. The thresholds are identical; the inputs aren't. Use CrUX p75 values when comparing to "passing".
  • INP vs FID. FID only measured the first interaction's delay, missing pages that get sluggish later. INP looks at the worst (or near-worst) interaction across the whole visit. INP replaced FID in Core Web Vitals on March 12, 2024.
  • CLS session window. Modern CLS sums shifts within a 5-second window with a 1-second gap, then reports the worst window — not the lifetime total. Late shifts after user input are excluded.
  • Mobile vs desktop. Google ranks pages using mobile-first CrUX data. Desktop scores can be much greener — don't let the desktop pass mask a failing mobile score.
